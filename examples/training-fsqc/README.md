# Example: Training-FSQC

This is the worked example from the CAISC 2026 paper. It demonstrates how a small structural-MRI quality-control corpus becomes a Glimmer research-object knowledge base, and how an AI agent reasons over it.

## What's here

- **`raw/`** — the source data: 3 subjects, 7 raters, 7 anatomical categories, FreeSurfer-derived QC metrics. Lives in the build script (`build_rokb.py`'s `RAW` constant) rather than a separate CSV; this is intentional so the example is self-contained.
- **`rokb/`** — generated by running `build_rokb.py`. 30 Glimmer nodes:
  - 3 `dataset` nodes (one per subject)
  - 1 `method` node (FreeSurfer 6.0 recon-all)
  - 1 `standard` node (the 0–5 ordinal rating scale)
  - 7 `rater` nodes (one per human researcher)
  - 18 `qc-artifact` nodes (one per rater-subject pair with non-missing ratings)

## What it demonstrates

The Training-FSQC corpus was built in 2016 as an inter-rater reliability calibration exercise: three representative subjects, seven researchers rating each independently across seven anatomical categories on a six-point ordinal scale.

For Glimmer, this corpus serves as a minimal-viable demonstration of:

1. **Heterogeneous entity types in a single graph.** Subjects, raters, methods, standards, and QC artifacts coexist with typed edges.
2. **The architectural-content distinction.** An agent given the bare schema (subjects + their CNR/Euler/talQC metrics + the rating-scale definition) performs differently from an agent that also reads peer raters' QC artifacts via graph edges.
3. **Inter-rater reliability as the comparator metric.** When the agent is itself just another node in the graph, the standard tool for measuring its agreement is Cohen's $\kappa$ — not accuracy against a single ground truth.

## Reproducing the paper's empirical demonstration

```bash
# 1. Build the RO-KB
python ../../glimmer/tools/build_rokb.py
#   Output: 30 nodes under ./rokb/

# 2. Run the agent (blind — metric-only graph)
OPENROUTER_API_KEY=$YOUR_KEY \
  python ../../glimmer/tools/agent.py \
    --model anthropic/claude-opus-4 \
    --output verdicts-blind.json

# 3. Run the agent (informed — adds peer QC-artifact edges)
OPENROUTER_API_KEY=$YOUR_KEY \
  python ../../glimmer/tools/agent.py \
    --model anthropic/claude-opus-4 \
    --informed \
    --output verdicts-informed.json

# 4. Score
python ../../glimmer/tools/score.py --verdicts verdicts-blind.json
python ../../glimmer/tools/score.py --verdicts verdicts-informed.json
```

Expected numbers (from the paper):

| Condition | Agent mean κ vs humans | Human mean κ | Δ |
|---|---|---|---|
| Blind | +0.066 | +0.478 | −0.412 |
| Informed | +0.583 | +0.478 | +0.105 |

## Why this corpus, why this size

Small. Three subjects, ~120 paired rating decisions per rater-pair after accounting for missing ratings. The κ statistic is well-defined at this scale because it's a pairwise measure, not a population estimator. Larger validation should target the full ADS Waves 1–3 manual edit records (N=141/106/97) or analogous corpora — see `docs/extending-the-example.md`.

## What's deliberately not here

- **No personal-graph data.** The rater names are author-acknowledged researchers; no participant identifiers or biological data are included.
- **No model weights.** The agent calls a frontier LLM via API at runtime; no model is bundled.
- **No raw imaging data.** The Training-FSQC corpus is metadata + ratings, not MRI volumes. The actual `.mgz`/`.nii.gz` files live with the parent ADS dataset under its data-use agreement.
